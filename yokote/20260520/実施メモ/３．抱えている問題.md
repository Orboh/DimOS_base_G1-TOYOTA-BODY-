# 3. 抱えている問題: G1 のカメラ・LiDAR が表示/利用できない

作成日: 2026-05-20

## 目的

`./dimensional-applications-g1/run_robot.sh` で G1 実機向けの DimOS agentic stack を起動したときに、G1 のカメラと LiDAR が使えず、可視化画面が真っ黒になる原因を調査し、解決したい。

## 現在の問題

G1 実機で以下を起動する。

```bash
cd /home/rad907/workspace/dimos-go2
./dimensional-applications-g1/run_robot.sh
```

起動後、G1 のカメラ・LiDAR 部分が正常に確認できない。

観測されている状態:

- カメラ表示が真っ黒になる
- LiDAR 表示/点群が確認できない
- その結果、視覚認識・空間認識・ナビゲーション系の機能が使えない可能性がある

## 比較対象

Go2 側では、カメラと LiDAR が確認できている。

つまり、DimOS 全体の可視化・カメラ/LiDAR表示の仕組みが完全に壊れているわけではなく、G1 stack 固有の接続・設定・topic・sensor source の問題である可能性が高い。

## 期待する状態

G1 実機で `run_robot.sh` を起動したときに、少なくとも以下が確認できる状態にしたい。

- G1 のカメラ画像が DimOS stream に流れる
- 可視化画面でカメラ画像が真っ黒ではなく表示される
- G1 の LiDAR または点群に相当するデータが DimOS stream に流れる
- 可視化画面で LiDAR/点群が確認できる
- LLM agent の視覚・空間系 skill が必要なセンサ入力を受け取れる

## 既にわかっていること

- G1 とは有線LANで疎通できていた
- G1 の腕ジェスチャー `RightHandUp` は、コントローラーで歩行モードにしたあと成功した
- そのため、G1 への WebRTC command 経路そのものは動いている
- 問題は、少なくとも腕コマンドではなく、カメラ/LiDAR/可視化/stream 側にある
- 起動時にローカルカメラ `/dev/video4` を開けない警告が出ていた

## 調査したい範囲

まず確認したいのは以下。

- G1 実機 stack が、カメラ入力をどこから取ろうとしているか
- G1 実機 stack が、LiDAR/点群入力をどこから取ろうとしているか
- Go2 stack と G1 stack で、センサ stream の構成がどう違うか
- G1 の `run_robot.sh` 起動時に、どの module がカメラ/LiDARを publish している想定か
- 可視化側が購読している topic と、G1 側が publish している topic が一致しているか
- G1 実機接続 `G1Connection` が sensor stream を publish しているか
- G1 の primitive/perceptive blueprint がローカルWebカメラ前提になっていないか
- 実機G1から WebRTC 経由でカメラ/LiDARを取る実装が存在するか

## 今回はまだやらないこと

このメモでは、まだ原因調査やコード修正は行わない。

ここでは、問題の要件と調査観点だけを整理する。

## 成功条件

調査・解決後の成功条件は以下。

- G1 実機起動時にカメラが黒画面ではなくなる
- G1 実機起動時に LiDAR/点群が確認できる
- Go2 では動いているが G1 では動かない理由を説明できる
- 必要であれば、G1 用 blueprint または connection の修正方針を明確にできる

## 現時点の原因仮説

技術に詳しい人から「ブループリントを確認してみて」とアドバイスをもらった。

ここでいう「ブループリント」は、DimOS の `blueprint` のことだと思われる。DimOS では、どの module を起動し、どの stream/topic を接続するかを定義する設計図に相当する。

現時点では、原因はかなり blueprint 側にありそう。

仮説:

```text
G1 実機 blueprint は、G1 本体のカメラ/LiDAR を WebRTC から publish していない。
代わりに、カメラはローカルPCの Webcam、
LiDAR/点群は外部 ROS/LCM topic 前提になっている可能性が高い。
```

### G1 側で見えている構成

G1 実機接続 `G1Connection` は、現状だと主に以下を担当している。

- `cmd_vel` を購読して G1 に移動指令を送る
- `publish_request()` で G1 に WebRTC request を送る
- 腕ジェスチャーなどの command はここから実行できる

一方で、`G1Connection` 自体には以下のような sensor output が見当たらない。

- `color_image: Out[Image]`
- `lidar: Out[PointCloud2]`
- `odom: Out[PoseStamped]`
- `camera_info: Out[CameraInfo]`

つまり、G1 実機接続 module は、Go2 のように G1 本体センサを DimOS stream に publish していない可能性がある。

該当ファイル:

```text
dimos/robot/unitree/g1/connection.py
```

### G1 のカメラはローカル Webcam 前提に見える

G1 の primitive blueprint では、実機時に `CameraModule` + `Webcam` を起動している。

該当ファイル:

```text
dimos/robot/unitree/g1/blueprints/primitive/uintree_g1_primitive_no_nav.py
```

ここでは `_create_webcam()` が `Webcam(camera_index=0, ...)` を返しており、G1 本体カメラではなく、PC 側のローカルカメラを開こうとしているように見える。

前回の起動ログに出ていた以下のエラーとも一致する。

```text
Failed to open camera 4
```

このため、G1 実機カメラが黒いというより、そもそも blueprint が G1 本体カメラを見ていない可能性が高い。

### G1 の LiDAR/点群は外部 topic 前提に見える

同じ G1 primitive blueprint では、LiDAR/点群らしき stream が以下の topic に割り当てられている。

```text
/lidar
/map
```

ただし、それらを `G1Connection` が publish しているようには見えない。

つまり、G1 の LiDAR/点群は、G1 本体 WebRTC 接続から直接来るのではなく、別の ROS/nav stack や外部 publisher が `/lidar` や `/map` に流す前提になっている可能性がある。

### Go2 側との違い

Go2 の `GO2Connection` は G1 と違い、sensor output を持っている。

該当ファイル:

```text
dimos/robot/unitree/go2/connection.py
```

Go2 では起動時に以下のような流れがある。

```text
connection.lidar_stream() -> self.lidar.publish
connection.video_stream() -> self.color_image.publish
connection.odom_stream()  -> self._publish_tf / self.odom.publish
```

そのため、Go2 ではカメラ・LiDAR が確認できる一方で、G1 では確認できない、という現象と一致する。

## G1 再接続後に確認したいこと

G1 と再び有線接続できたら、まず topic にデータが流れているかを確認する。

候補コマンド:

```bash
dimos topic echo /color_image
dimos topic echo /lidar
dimos topic echo /map
dimos topic echo /odom
```

確認したいこと:

- `/color_image` に画像が publish されているか
- `/lidar` に点群が publish されているか
- `/map` に点群/地図が publish されているか
- `/odom` に姿勢・位置情報が publish されているか
- publish されていない場合、どの module が本来 publish する想定なのか

次に、起動ログで以下を見る。

```bash
./dimensional-applications-g1/run_robot.sh
```

確認したいログ:

- `CameraModule` がどの camera index を開こうとしているか
- `Failed to open camera ...` が出るか
- `/lidar`, `/map`, `/odom` の transport 接続だけでなく、実データ publish があるか
- `G1Connection` が sensor stream を開始しているログがあるか

## 現時点の修正方針候補

まだ確定ではないが、解決方針としては以下のどれかになりそう。

1. G1 実機 blueprint を修正し、ローカル Webcam ではなく G1 本体カメラを使う
2. G1Connection に、G1 本体の camera / lidar / odom stream publish を追加する
3. G1 側で別途 ROS/nav stack を起動し、`/color_image`, `/lidar`, `/map`, `/odom` に流す
4. 既に存在する別の G1 blueprint に、実機センサ付きのものがあるか確認して切り替える

まずは、再接続後の topic 確認と起動ログ確認が必要。

## 現時点の結論

今回の実機ログと topic 確認から、原因は以下である可能性が高い。

```text
unitree-g1-agentic は、G1実機のカメラ/LiDARを直接publishする構成ではない。
カメラはPC側Webcam前提だが、PCに /dev/video* がないため画像が出ない。
LiDAR/点群は /lidar や /map topic 前提だが、現状それらをpublishするmoduleが動いていない。
```

したがって、G1でカメラ/LiDARを見たい場合は、次のどれかを検討する必要がある。

1. G1実機センサをpublishするmodule/blueprintを探して `unitree-g1-agentic` に組み込む
2. `unitree-g1-nav-onboard` のような LiDAR 用blueprintを基に agentic skill を合成する
3. G1Connection に Go2Connection 相当の camera/lidar/odom stream publish を実装する
4. まずは外部ROS/LCM publisherを起動し、`/color_image`, `/lidar`, `/map`, `/odom` に流す

## 2026-05-20 `models_clip.tar.gz` LFS 不足の解決

### 問題の根本原因

`SpatialMemory` は `ImageEmbeddingProvider(model_name="clip")` を使い、起動時に以下を実行する。

```python
model_id = get_data("models_clip") / "model.onnx"
self.model = ort.InferenceSession(str(model_id), providers=...)
```

`get_data("models_clip")` は `data/.lfs/models_clip.tar.gz` を Git LFS から pull しようとするが、
リモート (`https://github.com/Orboh/DimOS_base_G1.git/info/lfs`) が 404 を返すため失敗していた。

### 解決方法

LFS には頼らず、HuggingFace の `openai/clip-vit-base-patch32` を
`torch.onnx.utils.export` で直接 ONNX エクスポートして配置した。

実行した処理:

```python
from transformers import CLIPModel
import torch

model = CLIPModel.from_pretrained("openai/clip-vit-base-patch32")
model.eval()

class CLIPOnnxWrapper(torch.nn.Module):
    def __init__(self, clip_model):
        super().__init__()
        self.model = clip_model

    def forward(self, pixel_values, input_ids, attention_mask):
        outputs = self.model(
            pixel_values=pixel_values,
            input_ids=input_ids,
            attention_mask=attention_mask,
        )
        return outputs.image_embeds, outputs.text_embeds

torch.onnx.utils.export(
    CLIPOnnxWrapper(model),
    (pixel_values, input_ids, attention_mask),
    "data/models_clip/model.onnx",
    input_names=["pixel_values", "input_ids", "attention_mask"],
    output_names=["image_embeds", "text_embeds"],
    dynamic_axes={...},
    opset_version=14,
)
```

出力先:

```text
/home/rad907/workspace/dimos-go2/data/models_clip/model.onnx  (577 MB)
```

### 確認結果

```text
ImageEmbeddingProvider: Loaded CLIP model with providers: ['CPUExecutionProvider']
image embedding shape: (512,)
text embedding shape: (512,)
SpatialMemory init OK
```

`SpatialMemory` が `--disable spatial-memory` なしで起動できることを確認した。

### 補足

- `CUDAExecutionProvider` は現在の環境では利用不可（`AzureExecutionProvider, CPUExecutionProvider` のみ）
  - `onnxruntime-gpu` の CUDA バインディング未対応状態。UserWarning が出るが動作に問題はない
- LFS pointer ファイル (`data/.lfs/models_clip.tar.gz`) はそのまま残している
  - `data/models_clip/` ディレクトリが既に展開済みとして存在するため、`get_data()` は LFS pull をスキップする

## 2026-05-20 WebRTC 接続問題の調査（未解決・続きは次回）

### 問題の経緯

`unitree-g1-agentic` を起動すると `G1Connection` が WebRTC 接続を確立できず、
`Signaling State: have-local-offer` のまま進まない状態が続いていた。

事前確認として、UnitreeExplore アプリ（スマホ）が G1 に接続していたことが判明。
アプリを完全終了したが、その後も接続できなかった。

### 原因特定

調査の結果、以下の2つの問題が判明した。

**問題1: ICE candidates が SDP offer に含まれない**

`unitree_webrtc_connect` の `webrtc_driver.py` が、ICE gathering 完了を待たずに
SDP offer を G1 に送っていた。

- candidates=0 の offer → G1 が HTTP 504 を返す
- candidates=7〜15 の offer → G1 が HTTP 200 + SDP answer を返す（0.5秒で応答）

**修正済み** (`webrtc_driver.py` に ICE gathering 完了待ちを追加):

```python
ice_gathering_done = asyncio.Event()

@self.pc.on("icegatheringstatechange")
async def _on_ice_gather():
    if self.pc.iceGatheringState == "complete":
        ice_gathering_done.set()

if self.pc.iceGatheringState == "complete":
    ice_gathering_done.set()

try:
    await asyncio.wait_for(ice_gathering_done.wait(), timeout=5.0)
except asyncio.TimeoutError:
    logging.warning("ICE gathering did not complete within 5 s; proceeding anyway")
```
