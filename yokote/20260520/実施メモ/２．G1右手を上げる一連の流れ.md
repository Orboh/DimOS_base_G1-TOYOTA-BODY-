# G1 が「右手を上げて」に反応する一連の流れ

作成日: 2026-05-20

## 結論

G1 は、コントローラーで歩行モードに入れた状態にすると、DimOS 経由の自然言語指示から腕ジェスチャーを実行できた。

今回成功した指示は次のもの。

```bash
./dimensional-applications-g1/say.sh "右手を上げて"
```

内部的には、LLM がこの日本語指示を G1 の腕コマンド `RightHandUp` に変換し、`UnitreeG1SkillContainer.execute_arm_command()` 経由で G1 に WebRTC request を送った。

## 前提条件

### 1. G1 と有線LANで疎通している

PC 側は G1 と同じ `192.168.123.0/24` ネットワークに入っている。

確認時点では以下の状態だった。

```text
PC 有線IF: enp2s0
PC IP:    192.168.123.212/24
G1 IP:    192.168.123.161
```

`ping 192.168.123.161` は成功していた。

### 2. G1 をコントローラーで歩行モードにする

今回の重要な実機側条件。

G1 をコントローラーで歩行モードにしたことで、`RightHandUp` の腕ジェスチャーが実際に実行された。

DimOS 側にも `execute_mode_command("WalkMode")` という skill はあるが、今回の成功パターンでは、実機をコントローラー側で歩行モードにしてから自然言語指示を送った。

### 3. DimOS の G1 agentic blueprint を起動する

実機用スクリプト:

```bash
cd /home/rad907/workspace/dimos-go2
./dimensional-applications-g1/run_robot.sh
```

これは内部で次を実行する。

```bash
dimos run unitree-g1-agentic
```

`unitree-g1-agentic` には、G1 接続、MCP server、LLM agent、G1 skill container が含まれる。

## 全体フロー

```text
ユーザー
  |
  |  ./say.sh "右手を上げて"
  v
dimos agent-send
  |
  |  MCP tool: agent_send(message)
  v
McpServer
  |
  |  /human_input に publish
  v
McpClient
  |
  |  G1_SYSTEM_PROMPT と tool schema を見て LLM が判断
  v
LLM agent
  |
  |  execute_arm_command(command_name="RightHandUp")
  v
McpServer tools/call
  |
  |  RPC
  v
UnitreeG1SkillContainer.execute_arm_command()
  |
  |  topic="rt/api/arm/request"
  |  api_id=7106
  |  parameter.data=23
  v
G1Connection.publish_request()
  |
  |  Unitree WebRTC
  v
G1 実機
  |
  v
右手を上げる
```

## ステップ詳細

### 1. `say.sh` で自然言語を送る

`dimensional-applications-g1/say.sh` は薄いラッパーで、実体は `dimos agent-send`。

```bash
./dimensional-applications-g1/say.sh "右手を上げて"
```

内部では次を実行する。

```bash
dimos agent-send "右手を上げて"
```

### 2. `agent_send` が `/human_input` に流す

`dimos agent-send` は MCP server の `agent_send` tool を呼ぶ。

実装箇所:

```text
dimos/agents/mcp/mcp_server.py
```

`agent_send` は、受け取った文字列を LCM の `/human_input` に publish する。

```text
"右手を上げて" -> /human_input
```

### 3. `McpClient` が LLM に渡す

`McpClient` は `/human_input` を購読している。

実装箇所:

```text
dimos/agents/mcp/mcp_client.py
```

受信した文字列は `HumanMessage` として LLM agent の queue に入る。

このとき G1 用 blueprint では、Go2 用ではなく G1 用 system prompt が使われる。

```text
dimos/robot/unitree/g1/system_prompt.py
```

G1 用 prompt には、腕ジェスチャーとして `execute_arm_command` を使うこと、利用可能な command name に `RightHandUp` があることが書かれている。

### 4. LLM が tool call を選ぶ

「右手を上げて」という日本語指示に対して、LLM は次の tool call を選んだ。

```python
execute_arm_command(command_name="RightHandUp")
```

これは実行ログでも確認できた。

```text
tool call: execute_arm_command
args: {"command_name": "RightHandUp"}
```

### 5. `UnitreeG1SkillContainer` が G1 コマンドへ変換する

実装箇所:

```text
dimos/robot/unitree/g1/skill_container.py
```

`RightHandUp` は G1 arm command として定義されている。

```python
("RightHandUp", 23, "Raise only the right hand up.")
```

`execute_arm_command("RightHandUp")` は、次の request に変換される。

```python
topic = "rt/api/arm/request"
data = {
    "api_id": 7106,
    "parameter": {
        "data": 23,
    },
}
```

### 6. `G1Connection` が WebRTC 経由で送信する

実装箇所:

```text
dimos/robot/unitree/g1/connection.py
```

`UnitreeG1SkillContainer` は `_connection.publish_request(...)` を呼ぶ。

実際にログでは次の送信が確認できた。

```text
Publishing request to topic: rt/api/arm/request
data: {'api_id': 7106, 'parameter': {'data': 23}}
```

この request が Unitree WebRTC connection を通って G1 実機に届き、G1 が右手を上げる。

## 実行ログ上の成功ポイント

確認できた流れ:

```text
1. McpClient が MCP server から tools を取得
2. agent_send が「右手を上げて」を受信
3. LLM が execute_arm_command("RightHandUp") を選択
4. McpServer が execute_arm_command を実行
5. G1Connection が rt/api/arm/request に api_id=7106, data=23 を送信
6. tool result が "'RightHandUp' command executed successfully."
7. agent が「右手を上げました。」と返答
```

## 今回わかった注意点

- G1 側の状態が重要。今回、コントローラーで歩行モードにしたことで腕ジェスチャーが通った。
- `RightHandUp` は移動コマンドではなく arm command。`cmd_vel` ではなく `rt/api/arm/request` に送られる。
- `execute_arm_command` は `RightHandUp` など決められた command name を受け取る。
- `run_robot.sh --daemon` では MCP CLI から `localhost:9990` を見つけられない挙動があったため、確認時は foreground 起動のほうが扱いやすい。
- 起動時にローカルカメラ `/dev/video4` を開けない警告が出たが、今回の右手ジェスチャー自体はカメラを使わないため致命的ではなかった。
