## 2026-06-01 WebRTC video track の詳細確認

### 目的

`/color_image` が出ない原因が DimOS publish 側なのか、
Unitree WebRTC の video track 側なのかを切り分ける。

### SDP 確認

G1 から返った remote SDP には video track が含まれていた。

```text
m=video 9 UDP/TLS/RTP/SAVPF 101
a=sendonly
a=mid:1
a=msid:unitreeMediaStream unitreeVideoTrack
a=rtpmap:101 H264/90000
a=fmtp:101 level-asymmetry-allowed=1;packetization-mode=1;profile-level-id=42e01f
```

つまり、WebRTC negotiation 上は G1 が H264 video を送る形になっている。

### `unitree_webrtc_connect` 側の問題

インストール済み `unitree_webrtc_connect` の `webrtc_driver.py` では、
video track 受信時に callback を呼ぶ前に最初の frame を待っていた。

```python
if track.kind == "video":
    frame = await track.recv()
    await self.video.track_handler(track)
```

このため、最初の frame が来ない場合、DimOS 側の callback まで到達しない。

一時修正として、venv 内の以下を変更した。

```text
dimensional-applications-g1/.venv/lib/python3.12/site-packages/unitree_webrtc_connect/webrtc_driver.py
```

変更:

- `MediaStreamError` を catch
- video track では最初の `track.recv()` を待たずに `track_handler(track)` を呼ぶ

### DimOS wrapper 側の改善

対象:

```text
dimos/robot/unitree/connection.py
```

変更:

- `raw_video_stream()` に診断ログを追加
- `MediaStreamError` / その他例外をログ化
- 既に交渉済みの `pc.getReceivers()` から video track を拾う処理を追加
- `switchVideoChannel(True/False)` のログを追加

確認結果:

```text
Switching Unitree WebRTC video channel on
Video channel: on
Unitree WebRTC video track accepted kind=video ready_state=live
```

ここまで到達したため、DimOS 側は video track を読みに行けている。

ただし 40 秒待っても frame は 0 件だった。

```text
elapsed=40s frames=0 completed=False errors=[]
final frames=0 completed=False errors=[]
```

### 判断

現時点では、`/color_image` が出ない直接原因は以下と判断する。

```text
G1 main computer の WebRTC video track は negotiation され live 状態になるが、
実際の H264/RTP frame が送られてこない。
```

これは DimOS の topic 配線だけでは解決できない可能性がある。

3. `Device > Service status`: camera / videohub / perception / RealSense / LiDAR service の状態確認
4. `Device > Alarm information`: camera / sensor / communication error の確認
5. `Function`: vision / camera / inspection / AI 関連機能があるか確認
