# G1ロボットにおけるカメラ映像取得不可問題の追加調査報告

作成日: 2026-06-01

---

## 1. 要旨

2026-05-20 の調査で残課題となっていた、G1 実機カメラ映像を DimOS 上の `/color_image` に流せない問題について追加調査を行った。

調査の結果、**DimOS / WebRTC 接続の問題は大部分解消した一方で、G1 側でカメラ入力そのものが認識されていない可能性が高い**ことが判明した。

- **WebRTC 接続**: 前回未解決だった Data Channel Validation は成功し、G1 main computer との WebRTC 接続は確立できた。
- **DimOS topic**: LiDAR / FastLIO / odometry / costmap / camera_info は DimOS topic に流れていることを確認した。
- **カメラ映像**: WebRTC video track は negotiation され `live` 状態になるが、実際の frame は 0 件だった。
- **公式アプリ確認**: Unitree Explore アプリの `Go!` 画面でも正常なカメラ映像は見えず、ノイズ表示だった。
- **Jetson 確認**: G1 Jetson / Auxiliary PC 側でも `/dev/video*` が存在せず、`lsusb` / `dmesg` に RealSense / UVC Camera らしい認識ログはなかった。
- **DDS確認**: Unitree SDK2 で `rt/frontvideostream` を直接購読しても映像データは 0 件だった。

したがって、現時点では **DimOS 側のソフトウェア配線だけで解決できる段階ではなく、G1 本体側のカメラ搭載・物理接続・電源・ドライバ・サービス設定の確認が必要**である。

---

## 2. 背景・目的

### 背景

前回調査では、G1 の LiDAR / FastLIO 経路を `unitree-g1-agentic` に統合し、LiDAR 点群、地図、odometry が DimOS 上で流れることを確認した。

一方で、カメラについては以下が残課題だった。

- `G1Connection` に WebRTC 映像を DimOS stream へ流す処理を追加済み
- しかし、WebRTC の Data Channel Validation が完了せず、実機映像が `/color_image` に流れるか未確認

### 目的

今回の目的は以下。

- G1 との WebRTC 接続を確立する
- `/color_image` に G1 実機カメラ画像が流れるか確認する
- カメラが流れない場合、原因が DimOS 側か G1 本体側かを切り分ける
- LiDAR / FastLIO の動作を壊さず、カメラ問題の解決方針を明確にする

## 5. 結果

### 5.1 解決・確認できたこと

| 項目 | 結果 |
|------|------|
| G1 main computer への WebRTC 接続 | 成功 |
| Data Channel Validation | 成功 |
| LiDAR / FastLIO topic | 流れている |
| odometry / state_estimation / tf | 流れている |
| camera_info | 流れている |
| `/color_image` | 流れていない |
| WebRTC video track | SDP上存在し、receiver は live |
| WebRTC video frame | 0件 |
| DDS `rt/frontvideostream` | 0件 |
| Unitree Explore `Go!` | ノイズ表示 |
| Jetson `/dev/video*` | 存在しない |
| Jetson `lsusb` | RealSense / UVC Camera なし |
| Jetson `dmesg` | camera attach / `/dev/video` 生成なし |

### 5.2 現時点の結論

現時点では、`/color_image` が流れない原因は DimOS の topic 配線ではなく、**G1 側でカメラ入力が生成・publish されていないこと**にある可能性が高い。

最有力原因:

```text
G1 のカメラが Jetson / Auxiliary PC から認識されていない。
そのため videohub / frontvideostream / WebRTC video 経路へ映像を publish できていない。
```

## 6. 考察

前回調査では、G1 側のソフトウェア実装が Go2 に比べて不完全で、G1本体カメラを使う配線が不足していると判断していた。

今回の追加調査で、以下が新たに判明した。

1. WebRTC 接続自体は成功する
2. DimOS 側で video track を受け取るところまでは到達する
3. しかし、WebRTC でも DDS でも映像 frame が来ない
4. 公式アプリでも映像が正常に見えない
5. Jetson OS でもカメラデバイスが見えない

このため、今回の問題は以下の階層に分けられる。

| 階層 | 状態 |
|------|------|
| DimOS blueprint / topic 配線 | camera_info まで確認済み。`/color_image` は上流frameがなく未出力 |
| WebRTC transport | 接続・DataChannel は成功。video track はあるがframeが来ない |
| DDS / videohub | topicは存在するがframe publishなし |
| Jetson OS | `/dev/video*` なし、USB camera認識なし |
| 物理・本体構成 | 未確認。最優先で確認が必要 |

つまり、DimOS 側の修正だけを続けても、上流のカメラ入力が存在しない限り `/color_image` は流れない。

## 7. 課題・リスク

### 残課題

| 課題 | 詳細 | 優先度 |
|------|------|--------|
| G1 のカメラ搭載有無確認 | 対象G1が RealSense / depth camera 搭載構成か確認する | 高 |
| 物理接続確認 | カメラ本体、内部USBケーブル、Jetson側接続、USBハブ接続を確認する | 高 |
| Jetson側サービス確認 | camera / videohub / frontvideostream をpublishするサービスがあるか確認する | 高 |
| Unitreeサポート確認 | このG1個体で公式アプリ `Go!` に正常映像が出ないことを問い合わせる | 高 |
| DimOS側の暫定修正整理 | WebRTC video診断用の一時修正を正式に扱うか判断する | 中 |
| camera intrinsics | 映像が取得できた後、正確なカメラパラメータを確認する | 低 |

### リスク

`unitree_webrtc_connect` の venv 内ファイルに一時修正を入れている。

修正対象:

```text
dimensional-applications-g1/.venv/lib/python3.12/site-packages/unitree_webrtc_connect/webrtc_driver.py
```

この修正は依存パッケージ更新や venv 再作成で失われる可能性がある。
ただし、今回の結論としては、当該修正の有無に関わらず Jetson 側でカメラが認識されていないことが主要因と考えられる。

## 9. 開発グループへの相談事項

### 9.1 ハード確認の依頼

以下を確認したい。

1. 対象のG1がカメラ搭載モデルか
2. 搭載されている場合、カメラはどの計算機に接続される設計か
   - G1 main computer `192.168.123.161`
   - Jetson / Auxiliary PC `192.168.123.164`
3. Jetson側で `/dev/video*` が存在しない状態は正常か異常か
4. Unitree Explore `Go!` 画面でノイズ表示になるのは既知現象か
5. `rt/frontvideostream` の publisher が出ない場合、起動すべきサービスがあるか

### 9.2 ソフトウェア対応範囲の確認

現状、DimOS側でできる修正は以下まで進んでいる。

- WebRTC接続成功
- DataChannel成功
- video track受信地点まで到達
- camera_info publish
- LiDAR/FastLIO/odometry維持

しかし、Jetson OSにカメラデバイスが見えていないため、DimOSだけで解決するのは難しい。

開発グループとして、次をどちらに寄せるか判断したい。

- **案A**: G1本体・カメラ接続・Unitree側サービスの確認を先に進める
- **案B**: DimOS側で別入力ソース（外付けUSBカメラ、RTSP、ROS image topic等）を一時的に用意して代替する
- **案C**: Unitreeサポートに問い合わせ、G1カメラ構成と `frontvideostream` の起動条件を確認する

現時点では、案Aと案Cを優先するのが妥当と考える。
